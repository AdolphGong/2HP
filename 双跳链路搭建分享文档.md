# 双跳链路搭建分享文档
为了稳健使用Claude+Codex订阅，搭建此专属双跳链路，在你的任何端上（Windows、Mac、Mobile、Web）使用此双跳链路，可以保证你访问ChatGPT和Anthropic的出口IP始终在美国，一定程度避免频繁切IP被检测或者普通机场VPN批量被ban的问题。

这是一份可分享的搭建指南：不包含任何真实服务器密码、代理密码或个人节点地址。照着做可以搭出一条“本地客户端 -> 首段 VPS -> 出口节点”的双跳链路。

## 目标架构

```text
本地 Clash Verge
  -> 第一跳：离自己近的 VPS，跑 Hysteria2 入站和 sing-box 分流
  -> 第二跳：目标地区的 SOCKS5 出口
  -> 目标网站
```

第一跳负责“接入和分流”，第二跳负责“最终落地”。普通流量可以本地直连或从第一跳直连出去，只有需要指定落地地区的域名才转发到第二跳。对 AI/编程域名建议采用失败关闭策略：规则显式命中双跳，代理组里不放 `DIRECT`。

## 先准备什么

| 项目 | 建议 |
| --- | --- |
| 本地客户端 | Clash Verge 或兼容 Clash Meta 内核的客户端 |
| 第一跳 VPS | Ubuntu/Debian，离你近，UDP 稳定，公网 IPv4 固定 |
| 第二跳出口 | SOCKS5，目标网站友好的地区和网络类型 |
| 服务端程序 | sing-box |
| 第一跳协议 | Hysteria2/UDP |
| 第一跳端口 | 随机高位 UDP 端口，例如 `20000-60000` |

## 节点怎么选

### 第一跳选型

第一跳不一定要在目标网站所在国家，它更像“入口加速器”。选择标准：

- 离自己近：大陆用户常见选择是新加坡、日本、香港、韩国等。
- UDP 质量好：Hysteria2 走 QUIC/UDP，运营商到机房的 UDP 丢包越少越好。
- 公网 IP 稳定：避免频繁换 IP，否则客户端和第二跳白名单都要改。
- 云厂商安全组好管理：必须能放行自定义 UDP 端口。
- 带宽够用：第一跳会承接所有客户端入口流量。

不要只看延迟最低。更重要的是高峰期丢包、绕路、晚间稳定性。

### 第一跳资源基线

第一跳不要只看延迟和带宽，也要看内存和云盘 IOPS。Hysteria2 + sing-box 本身不重，但低配节点在连接抖动、DNS 查询、系统更新或 page cache 被回收时，可能出现“CPU 不高但整机假死”的状态。

建议基线：

- 内存至少 1G，并配置 1G swap；预算允许时选 2G 内存更稳。
- 云盘不只看容量，还要看 IOPS；低 IOPS 云盘在内存压力下容易被读 IO 风暴拖死。
- 长期运行时 `sing-box` 日志级别用 `warn`，排障时再临时切到 `info`。
- 看到 `systemd-journald: Under memory pressure, flushing caches.`、`load average` 很高、`blocked` 很多、`sar -d` 里 `rkB/s` 高而 `wkB/s` 很低时，优先按内存压力和读 IO 风暴排查，不要误判为日志写爆。

### 第二跳选型

第二跳才决定目标网站看到的出口 IP。选择标准：

- 地区匹配目标：如果目标服务希望美国环境，第二跳就选美国。
- IP 类型干净：AI、风控敏感站点通常更偏好住宅/ISP/干净静态 IP。
- 支持账号密码：SOCKS5 必须有认证。
- 支持白名单：只允许第一跳 IP 访问，防止出口被扫到后被滥用。
- 尽量静态：频繁换 IP 会影响账号风控、登录状态和可用性。

如果第二跳是自有 VPS，可以直接在第二台机器上跑 SOCKS5；如果是第三方住宅代理，则在第一跳配置 SOCKS5 出站即可。

### 节点地址怎么选

- 客户端 `server` 填第一跳地址，不填第二跳地址。
- 有自己的域名时，推荐把域名 A 记录指向第一跳，并申请真实 TLS 证书。
- 没有域名时，也可以用第一跳 IP + 自签证书，但客户端要开启跳过证书验证。
- 第二跳地址只写在第一跳的 sing-box 配置里，不暴露给本地客户端。
- 第二跳如果是自建 SOCKS5，防火墙只放行第一跳 IP。

端口建议随机高位，不建议用太常见的 `443`、`8443`、`1080`。随机高位端口不是安全措施本身，但能少一些无意义扫描。

## 占位符

后面的命令和配置里会用这些占位符，先替换成你自己的值。

| 占位符 | 含义 | 示例 |
| --- | --- | --- |
| `<FIRST_IP>` | 第一跳 VPS 公网 IP | `203.0.113.10` |
| `<FIRST_PORT>` | 第一跳 Hysteria2 UDP 端口 | `25000` |
| `<HY2_PASSWORD>` | 第一跳 Hysteria2 密码 | 随机长字符串 |
| `<TLS_NAME>` | 证书 CN/SNI | 有域名填域名；无域名可填一个固定名称 |
| `<EXIT_IP>` | 第二跳 SOCKS5 地址 | `198.51.100.20` |
| `<SOCKS_PORT>` | 第二跳 SOCKS5 端口 | `30080` |
| `<SOCKS_USER>` | 第二跳 SOCKS5 账号 | `user1` |
| `<SOCKS_PASS>` | 第二跳 SOCKS5 密码 | 随机长字符串 |

生成密码可以用：

```bash
openssl rand -base64 24
```

## 第一步：安装 sing-box

两台 VPS 都执行：

```bash
sudo apt-get update
sudo apt-get install -y curl ca-certificates openssl iptables-persistent
curl -fsSL https://sing-box.app/install.sh | sudo sh
```

安装后常用命令：

```bash
sudo systemctl enable sing-box
sudo systemctl start sing-box
sudo systemctl status sing-box
sudo journalctl -u sing-box --output cat -f
```

## 第一步补充：低配 VPS 基础加固

低配第一跳建议在安装服务后先做三件事：加 swap、限制 journald、生产环境降低 sing-box 日志级别。这样可以避免内存压力触发 page cache 回收，进而把云盘 IOPS 打满。

```bash
# 增加 1G swap
sudo fallocate -l 1G /swapfile
sudo chmod 600 /swapfile
sudo mkswap /swapfile
sudo swapon /swapfile
echo '/swapfile none swap sw 0 0' | sudo tee -a /etc/fstab

# 设置 swappiness=10；如果系统已有 swappiness=0，需要一并改掉
sudo sed -i.bak -E 's/^[[:space:]]*vm\.swappiness[[:space:]]*=.*/vm.swappiness=10/' /etc/sysctl.conf
sudo sed -i.bak -E 's/^[[:space:]]*vm\.swappiness[[:space:]]*=.*/vm.swappiness=10/' /etc/sysctl.d/99-sysctl.conf 2>/dev/null || true
echo 'vm.swappiness=10' | sudo tee /etc/sysctl.d/99-vpnroot.conf
sudo sysctl -w vm.swappiness=10
sudo sysctl --system

# 限制 journald 占用和突发日志
sudo mkdir -p /etc/systemd/journald.conf.d
sudo tee /etc/systemd/journald.conf.d/99-vpnroot.conf >/dev/null <<'EOF'
[Journal]
SystemMaxUse=100M
RuntimeMaxUse=32M
RateLimitIntervalSec=30s
RateLimitBurst=200
EOF
sudo systemctl restart systemd-journald
```

`sing-box` 配置在调试阶段可以用 `info`；稳定后建议改为 `warn`。本地 Clash 仍然可以观察域名是否命中本地规则和双跳节点；只有当你要在 VPS 上确认某个域名是否转发到了第二跳 SOCKS5 时，才需要临时切回 `info` 看 `outbound/socks` 日志。

## 第二步：配置第二跳 SOCKS5

如果第二跳是第三方 SOCKS5 服务，跳过本节，直接拿到 `<EXIT_IP>`、`<SOCKS_PORT>`、`<SOCKS_USER>`、`<SOCKS_PASS>` 即可。

如果第二跳是自己的 VPS，在第二台机器写入 `/etc/sing-box/config.json`：

```json
{
  "log": {
    "level": "warn",
    "timestamp": true
  },
  "inbounds": [
    {
      "type": "socks",
      "tag": "socks-in",
      "listen": "0.0.0.0",
      "listen_port": <SOCKS_PORT>,
      "users": [
        {
          "username": "<SOCKS_USER>",
          "password": "<SOCKS_PASS>"
        }
      ]
    }
  ],
  "outbounds": [
    {
      "type": "direct",
      "tag": "direct"
    }
  ],
  "route": {
    "final": "direct"
  }
}
```

检查并启动：

```bash
sudo sing-box check -c /etc/sing-box/config.json
sudo systemctl restart sing-box
sudo ss -lntup | grep <SOCKS_PORT>
```

只允许第一跳访问 SOCKS5 端口：

```bash
sudo iptables -A INPUT -p tcp -s <FIRST_IP> --dport <SOCKS_PORT> -j ACCEPT
sudo iptables -A INPUT -p tcp --dport <SOCKS_PORT> -j DROP
sudo netfilter-persistent save
```

如果云厂商还有安全组，也在安全组里只允许 `<FIRST_IP>` 访问 TCP `<SOCKS_PORT>`。

### 第二跳基础加固

第二跳职责比第一跳简单，但如果是 1G 内存的小机器，也建议做同样的低成本保险。即使面板显示“数据磁盘 0GB”，也可以把 swap 文件放在系统盘 `/` 上；先确认系统盘空间足够。

```bash
df -h /
free -h
swapon --show
```

如果 `/` 至少有 2G 可用，增加 1G swap：

```bash
sudo fallocate -l 1G /swapfile || sudo dd if=/dev/zero of=/swapfile bs=1M count=1024 status=progress
sudo chmod 600 /swapfile
sudo mkswap /swapfile
sudo swapon /swapfile

grep -q '^/swapfile ' /etc/fstab || echo '/swapfile none swap sw 0 0' | sudo tee -a /etc/fstab

sudo sed -i.bak -E 's/^[[:space:]]*vm\.swappiness[[:space:]]*=.*/vm.swappiness=10/' /etc/sysctl.conf
sudo sed -i.bak -E 's/^[[:space:]]*vm\.swappiness[[:space:]]*=.*/vm.swappiness=10/' /etc/sysctl.d/99-sysctl.conf 2>/dev/null || true
echo 'vm.swappiness=10' | sudo tee /etc/sysctl.d/99-vpnroot.conf

sudo sysctl -w vm.swappiness=10
sudo sysctl --system

swapon --show
free -h
cat /proc/sys/vm/swappiness
```

限制 journald，避免异常连接或系统压力时日志继续放大 IO 压力：

```bash
sudo mkdir -p /etc/systemd/journald.conf.d
sudo tee /etc/systemd/journald.conf.d/99-vpnroot.conf >/dev/null <<'EOF'
[Journal]
SystemMaxUse=100M
RuntimeMaxUse=32M
RateLimitIntervalSec=30s
RateLimitBurst=200
EOF

sudo systemctl restart systemd-journald
```

如果执行 `sudo` 时看到 `unable to resolve host`，补一条本机 hostname 到 `/etc/hosts`：

```bash
grep -q "$(hostname)" /etc/hosts || echo "127.0.1.1 $(hostname)" | sudo tee -a /etc/hosts
sudo true
```

第二跳稳定后也建议把 `sing-box` 日志级别设为 `warn`：

```bash
sudo cp -a /etc/sing-box/config.json /etc/sing-box/config.json.bak.$(date +%F-%H%M%S)

sudo python3 -c 'import json,pathlib
p=pathlib.Path("/etc/sing-box/config.json")
data=json.loads(p.read_text())
data.setdefault("log", {})
data["log"]["level"]="warn"
data["log"]["timestamp"]=True
p.write_text(json.dumps(data, ensure_ascii=False, indent=2) + "\n")'

sudo sing-box check -c /etc/sing-box/config.json
sudo systemctl restart sing-box
grep -A5 -n '"log"' /etc/sing-box/config.json
```

验证目标状态：`swapon --show` 能看到 `/swapfile 1024M`，`cat /proc/sys/vm/swappiness` 输出 `10`，`grep -A5 -n '"log"' /etc/sing-box/config.json` 里显示 `"level": "warn"`。

## 第四步：配置 Clash Verge

在本地新建一个 Clash Profile：

```yaml
mixed-port: 7897
allow-lan: false
mode: rule
log-level: info

proxies:
  - name: DOUBLE-HOP
    type: hysteria2
    server: <FIRST_IP>
    port: <FIRST_PORT>
    password: <HY2_PASSWORD>
    sni: <TLS_NAME>
    skip-cert-verify: true

proxy-groups:
  - name: WORK-AI
    type: select
    proxies:
      - DOUBLE-HOP

rules:
  # AI/API/AI 编程流量失败关闭：WORK-AI 不包含 DIRECT。
  - DOMAIN-SUFFIX,openai.com,WORK-AI
  - DOMAIN-SUFFIX,chatgpt.com,WORK-AI
  - DOMAIN-SUFFIX,anthropic.com,WORK-AI
  - DOMAIN-SUFFIX,claude.ai,WORK-AI
  - DOMAIN-SUFFIX,googleapis.com,WORK-AI
  - DOMAIN-SUFFIX,gemini.google.com,WORK-AI
  - DOMAIN-SUFFIX,githubcopilot.com,WORK-AI
  - DOMAIN-SUFFIX,copilot-proxy.githubusercontent.com,WORK-AI
  - DOMAIN-SUFFIX,cursor.com,WORK-AI
  - DOMAIN-SUFFIX,windsurf.com,WORK-AI
  - DOMAIN-KEYWORD,copilot,WORK-AI
  - DOMAIN-KEYWORD,codex,WORK-AI

  # 普通未命中流量本地直连，避免第一跳异常拖垮日常网络。
  - MATCH,DIRECT
```

Clash Verge 主界面建议：

- 系统代理：开启。
- TUN：关闭。
- 模式：Rule。
- Profile：选中新建的配置。

如果你使用真实域名和可信证书，可以把 `skip-cert-verify` 改为 `false` 或删除。

## 第五步：验证

本地检查第一跳是否可用：

```powershell
curl.exe -x http://127.0.0.1:7897 https://www.cloudflare.com/cdn-cgi/trace --max-time 10
```

检查目标 AI 流量是否落在第二跳：

```powershell
curl.exe -x http://127.0.0.1:7897 https://chatgpt.com -I --max-time 10
```

看 `CF-RAY` 后缀：

- 第二跳目标地区：分流成功。
- 第一跳地区：AI 域名没有进 SOCKS5，需要看第一跳 sing-box 路由和日志。

在第一跳看日志：

```bash
sudo journalctl -u sing-box --output cat -f
```

在第二跳看 SOCKS5 是否有流量：

```bash
sudo journalctl -u sing-box --output cat -f
```

## 常见问题

### 客户端连接不上第一跳

- 检查第一跳安全组是否放行 UDP `<FIRST_PORT>`。
- 检查第一跳 `systemctl status sing-box`。
- 检查 Clash 配置里的 IP、端口、密码、SNI 是否一致。
- 自签证书场景下确认 `skip-cert-verify: true`。

### 第一跳能连，但 AI 没走第二跳

- 检查第一跳配置里的 `domain_suffix` 和 `domain_keyword`。
- 检查第二跳 SOCKS5 地址、账号、密码。
- 检查第二跳防火墙是否放行 `<FIRST_IP>`。
- 检查是否被 QUIC/UDP 绕过，必要时继续保留 AI 域名 UDP block 规则。
