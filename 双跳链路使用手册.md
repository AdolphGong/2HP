# 双跳链路使用手册

这是一份脱敏后的日常使用手册，只描述操作流程和排障思路，不包含真实服务器 IP、SSH 用户、系统密码、代理账号密码、机器名或本地私有路径。所有需要替换的信息都统一写成 `<PLACEHOLDER>`。

## 使用前确认

- 遵守所在地法律法规、云服务商政策和目标服务条款。
- 只把这条链路用于合法的学术研究、开发测试和稳定访问场景。
- 不要把真实节点信息、账号密码或可直接导入的个人配置发到公开仓库、聊天群或截图里。
- 如果任何真实凭据曾经公开过，先轮换凭据，再清理公开内容。

## 日常使用流程

1. 确认第一跳 VPS 和尾段节点都处于可用状态。
2. 打开 Clash Verge、Mihomo 或移动端兼容客户端。
3. 导入对应模板，并替换 `<FIRST_IP>`、`<FIRST_PORT>`、`<HY2_PASSWORD>`、`<TLS_NAME>` 等占位符。
4. 选择 `Rule` 模式，让普通流量按规则直连或走默认策略，让目标流量进入双跳链路。
5. 用响应头、客户端日志和第一跳日志确认目标流量确实进入预期路径。

## PC 客户端

PC 端建议使用 Clash Verge / Clash Verge Rev 或其它兼容 Mihomo 内核的客户端。

- 导入 `PC客户端配置模板.md` 或 `examples/clash-verge.template.yaml`。
- 把所有 `<PLACEHOLDER>` 替换成自己的真实值。
- 保持工作类、AI 类和开发类域名走双跳代理组。
- 普通未命中的流量默认 `DIRECT`，减少不必要的绕路。
- 如果还有本地媒体代理，可以保留模板中的本地上游；没有就删除对应代理组和规则。

## Android / iOS 移动端

- 移动端优先使用专门的 Android / iOS 配置模板。
- 移动端通常没有本机媒体上游，模板会更简单。
- 导入后先只测试少量目标域名，再逐步扩展规则。
- 如果客户端不支持某些规则集语法，保留 `DOMAIN-SUFFIX` 和 `DOMAIN-KEYWORD` 这类基础规则即可。

## 验证链路

在 PC 端可以先用本地 HTTP 代理端口检查外部访问是否正常：

```powershell
curl.exe -x http://127.0.0.1:7897 https://www.cloudflare.com/cdn-cgi/trace --max-time 10
```

再测试目标站点是否能通过代理访问：

```powershell
curl.exe -x http://127.0.0.1:7897 https://example.com -I --max-time 10
```

验证时重点看三件事：

- 客户端日志是否命中预期规则和代理组。
- 第一跳日志是否收到来自本机客户端的连接。
- 目标网站看到的出口是否来自尾段节点，而不是本机或第一跳。

## 常见切换场景

| 场景 | 建议 |
| --- | --- |
| 工作 / AI / 开发 | 使用双跳代理组，保持出口稳定。 |
| 普通浏览 | 默认直连或按本地习惯分流。 |
| 媒体或娱乐 | 可以单独放到媒体代理组，避免和工作链路混用。 |
| 临时排障 | 暂时把指定域名加入测试规则，确认后再整理进长期规则。 |

## 排障清单

### 客户端连不上第一跳

- 检查 `<FIRST_IP>`、`<FIRST_PORT>`、`<HY2_PASSWORD>`、`<TLS_NAME>` 是否一致。
- 检查 VPS 安全组和系统防火墙是否放行 UDP `<FIRST_PORT>`。
- 检查客户端是否启用了跳过自签证书验证，或是否使用了真实可信证书。
- 检查第一跳服务是否正常运行。

### 第一跳能连，但目标流量没有走尾段

- 检查第一跳路由规则里的 `domain_suffix`、`domain_keyword` 是否覆盖目标域名。
- 检查尾段 SOCKS5 地址、端口、账号和密码。
- 检查尾段是否只允许第一跳 IP 访问。
- 检查客户端代理组里是否误放了 `DIRECT`。

### 访问不稳定

- 先区分是本机到第一跳不稳定，还是第一跳到尾段不稳定。
- 关注 UDP 丢包、云盘 IO、内存压力和系统负载。
- 排障时临时提高日志级别，恢复正常后再调回较低级别。
