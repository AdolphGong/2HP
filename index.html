Run checklist

Traffic policy at a glance

A double-hop chain is most useful when each traffic class has a clear destination and failure mode.

Traffic class Route Rule intent Failure behavior
AI and API domains DOUBLE HOP Pin sensitive target domains to the controlled exit region. FAIL CLOSED
Developer tooling DOUBLE HOP Keep coding assistants and package/dev surfaces on the same controlled path. FAIL CLOSED
Media traffic LOCAL UPSTREAM Avoid burning the double-hop exit for high-volume streaming. Use a separate client or bypass.
Ordinary unmatched traffic DIRECT Keep daily browsing resilient when the proxy chain is degraded. Direct fallback.

Setup flow

Use the guide as a controlled sequence instead of copying a private configuration wholesale.

1

Pick the hops

Choose a nearby UDP-stable first hop and a target-region SOCKS5 exit.

2

Install sing-box

Run Hysteria2 inbound on the first hop and SOCKS5 outbound policy.

3

Lock rules

Put sensitive domains before broad fallback rules and avoid DIRECT in fail-closed groups.

4

Import templates

Replace placeholders locally, then import the desktop or Android YAML.

5

Verify route

Check response headers, client logs, and first-hop logs before trusting the path.

Share the pattern, never the live credentials.

2HP is designed as a public documentation repository. Anything that can connect to your actual infrastructure belongs outside Git or in encrypted storage.

  • Keep operational notes, real YAML files, SSH users, and passwords ignored by Git.
  • Use placeholders such as <FIRST_IP>, <HY2_PASSWORD>, and <SOCKS_PASS> in public files.
  • Scan staged content before publishing, not only the working directory.
  • Rotate any credential that has ever reached a public remote, even briefly.
